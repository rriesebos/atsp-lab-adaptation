Web Cache Deception

What?

In 2018, a study by Guo et al. showed that about 74% of the 1000 most popular websites by Alexa ranking are using a Content Delivery Network (CDN). These CDNs are a proxy in between the client (such as web browsers) that are geographically close to the user and provide various benefits. On such benefit is to reduce latency and network traffic by caching static files.

In 2017, a vulnerability was discovered by Omer Gil as explained on his Blog called Web Cache Deception. Web Cache Deception allows an attacker to steal highly confidential information of a victim, whenever the CDN has cache rules that are too aggressive.

How?

First, the attacker needs to find a vulnerable website by trying to modify the URL. There are two conditions that must hold for this URL, in order to perform an attack:

  • The intermediate proxy with caching must think the URL is a static file and will cache the response from the web server.
  • The URL must be interpreted as a regular URL by the web server, and thus, return personal information on the webpage.

Let's say the attacker is looking for performing this attack on the following fictional URL: somebank.com/profile. The attacker will modify the URL to the following: somebank.com/profile/img.png and notices it shows the initial profile webpage. While opening an incognito browser, the user sees the exact same webpage without being logged in!

Hence, the attacker sends a slightly different URL (to bypass cache) to the victim. The victim opens the URL, but sees a regular webpage. The user has no reason for suspicion, because there is nothing odd about the webpage (besides the slight URL modification).

The attacker checks back on the same URL that the attacker sent to the victim, to see if it has been opened, and thus, cached yet. If so, the attacker now sees the same HTML-content as the victim when the victim is logged in!

Consequences

There are two major consequences.

First and most obviously, every information that the web server shows to the user is now in the hands of the attacker. It depends on the website and webpage, but examples include username, mail address, home address, telephone number, bank details, order history, chat messages, credit card information and anything else you can think of.

There is also a second consequence that perhaps occurs less frequently, but could be even more dangerous. In case the web server also embeds any sort of tokens in the HTML DOM, these can be stolen by the attacker as well. The first example would be when the page includes a form with a token linked to the user, allowing the attacker to submit a form on the user's behalf. A second example could be that the webpage embeds cookies / session tokens in the page. Then, an attacker can simply steal the session token, and depending on the security checks of the website, the attacker could be logged in on the user's account.
